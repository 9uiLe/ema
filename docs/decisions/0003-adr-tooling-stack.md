## Context {#context}

[ADR-0002](0002-adr-source-format.html) は「正本＝制約付き Markdown／人間レビュー用 HTML は **外部依存ゼロ** のジェネレータで生成」を決め、`tools/adr-gen.mjs`・`adr-lint.mjs`・`adr-view.mjs` として実装された。残っていた問いは **設計判断ではなく実装品質** — このツール群を **個人が中長期で1人保守する** とき、現状の構成（~425 行の自作 Markdown パーサ、テストなし）は最適か。

quality-architecture スキル（ISO/IEC 25010:2023）で評価し、利用形態と依存方針を確定した：

| 前提 | 値 |
|---|---|
| 利用形態 | 個人・ローカルのみ（手元で `node tools/...`、HTML は閲覧用の一時生成、外部公開・チーム配布なし） |
| 依存方針 | 検証付き限定許容（既定ゼロ依存、明確に優位な成熟ライブラリのみ監査の上で検討） |

### 非最適点は「スタック」ではなく「試験性」だった {#diagnosis}

Node 標準のみという選択は、個人・ローカル・閲覧用途には適切（性能効率性・移植性は要求が低く、ここに依存を足すのは過剰設計）。問題は別にあった。評価中、自作 `inline()` のコードスパン保護センチネルが **裸の数字 ` 3 ` を誤ってコードスパン化** する欠陥を実証で確認した（`Node version 3` → `version<code>undefined</code>`、別コードスパン本文との相互汚染も発生）。テストが無いためこの種の劣化が静かに蓄積していた。

> [!NOTE] 「ADR 数に比例」の本当の意味
> 当初の懸念は「HTML 生成が ADR 数に比例してスケールするか」だった。だが O(n) 生成＋差分スキップ済みで、個人スケール（多くて数十本）では **性能は非問題**。本当に ADR 数に比例して効くのは *スループットではなく、新しい ADR ごとに未テストのパーサが新しいエッジケースを踏み、正確性が劣化するリスク*。これは性能ではなく **試験性・機能正確性** の問題。

### 鍵となる洞察：試験性はゼロ依存のまま埋まる {#key-insight}

「自作パーサの保守・正確性が不安」がライブラリ導入の主動機になりうる。だが **`node:test` は Node 標準** であり、依存を1つも増やさずにユニットテストを書ける。つまりライブラリ導入の主目的は **依存追加なしで達成でき**、ライブラリは install 摩擦と供給網リスクを足すだけになる。さらに、この ADR ツールの差別化価値は Markdown 解析そのものではなく **ADR 固有の意味づけ**（固定スロット・反証スロット・cmp-mark 比較表・推奨列 `[*]`・デザインシステム HTML）にある。解析は意図的に小さい制約付き文法であり、フル CommonMark は要らない（小さく保つこと自体が流し読み防止の機能）。

## Decision Drivers {#drivers}

- **供給網安全** — 外部依存の攻撃面。ユーザー方針は検証付き限定許容（既定ゼロ依存）。
- **試験性** — 機能正確性を回帰テストで担保できるか。承認は描画でするため、描画の正しさが運用の土台。
- **install 摩擦の小ささ** — 個人・ローカルで `git clone` 後すぐ `node tools/...` が動くか。
- **制約付き文法との適合** — 意図的に小さい文法を、過不足なく扱えるか。
- **保守性（seam 分離・DRY）** — 変換ロジックがテスト可能な継ぎ目に分離され、規約が単一表現か。

> [!TIP] 試験性が分岐を消す
> 「ライブラリ導入 vs 自作」という本物のトレードオフは、試験性をどう確保するかで決まる。`node:test` でゼロ依存のまま確保できると分かった時点で、ライブラリ導入の主目的が依存なしで満たされ、トレードオフは **発生しない**。

## Considered Options {#options}

### Option A — ゼロ依存維持 ＋ node:test で試験性を確保（推奨） {#option-a}

Node 標準のみを維持。変換の純粋関数を `tools/lib/adr-render.mjs` に seam として抽出し、`tools/adr-render.test.mjs` が `node:test` で検証する。`adr-gen.mjs`・`adr-lint.mjs`・`adr-view.mjs` はこの共有モジュールを使う（frontmatter 解析・Status 値・esc を単一表現化）。

- **長所**: 供給網ゼロ・install 摩擦ゼロ・制約付き文法に過不足なく適合。試験性を依存追加なしで獲得。seam 分離と DRY で保守性向上。自作入力（自分で書く ADR）なので攻撃面は実質低い。
- **短所**: Markdown 文法を拡張するときは自分で実装＋テストする必要がある。CommonMark 準拠ではない（意図的）。

### Option B — 成熟 Markdown ライブラリへ移行 {#option-b}

`markdown-it` 等の成熟パーサに解析を委ね、ADR 固有の意味づけ（callout・cmp-mark）はプラグイン／後処理で実装する。

- **長所**: 解析の正確性・多機能 Markdown への拡張余地が高い。自作の解析コードを減らせる。
- **短所**: 推移的依存が増え、**ゼロ依存・供給網安全と install 摩擦** を悪化させる。導入の主目的（正確性・保守性）は Option A が依存なしで達成済みで、利点が薄い。制約付き文法には過剰。

### Option C — 成熟 SSG へ全面移行 {#option-c}

静的サイトジェネレータ（例：既製のドキュメントツールチェーン）へ全面的に載せ替える。

- **長所**: 自作コードを最小化。公開・テーマ・検索などが揃う。
- **短所**: 個人・ローカル・閲覧用途には **過剰設計**。依存とビルドの複雑さが跳ね上がり、デザインシステムと反証スロット等の固有要件を載せ替えるコストが大きい。供給網・install 摩擦が最悪。

| 判断軸 | A ゼロ依存＋node:test [*] | B Markdown ライブラリ | C SSG 全面移行 |
|---|---|---|---|
| 供給網安全（攻撃面の小ささ） | yes | mid | no |
| 試験性の確保（正確性担保） | yes | yes | yes |
| install 摩擦の小ささ（個人ローカル） | yes | mid | no |
| 制約付き文法との適合 | yes | mid | no |
| 自作コード量の少なさ | mid | yes | yes |
| 多機能 Markdown への拡張余地 | no | yes | yes |

## Decision {#decision}

> [!SUCCESS] 採用：Option A — ゼロ依存維持 ＋ node:test で試験性を確保
> ADR ツール群の技術スタックは **Node 標準モジュールのみ** を維持する。変換の純粋関数を副作用のない `tools/lib/adr-render.mjs` に分離し、`tools/adr-render.test.mjs`（`node:test`、**依存ゼロ**）で回帰テストする。`adr-gen.mjs` は薄い CLI（ファイル IO＋引数）に、`adr-lint.mjs`／`adr-view.mjs` は frontmatter 解析・Status 値・esc を lib と共有する（DRY）。
>
> B（Markdown ライブラリ）に対して A を採る決め手は、**試験性が依存ゼロで確保できる** こと — ライブラリ導入の主目的（正確性・保守性）が依存追加なしで満たされ、ライブラリは install 摩擦と供給網リスクだけを足すため。C（SSG）は個人・ローカル・閲覧用途に対し過剰設計。
>
> 2026-06-06 承認。実装済：seam 抽出、回帰テスト16件、frontmatter 単一表現化、実証済み `inline()` 欠陥の修正（センチネルを私用面 `U+E000/E001` 化）、`tools/` に残っていた旧採番（0007/0008）の解消。`gen --check` 緑（出力不変＝挙動保存）・`node --test` 16/16・lint ERROR 0。

## Consequences {#consequences}

### Positive {#consequences-positive}

- 機能正確性を回帰テストで固定でき、新しい ADR がエッジケースを踏んでも劣化を検知できる。
- 供給網ゼロ・install 摩擦ゼロを維持。CI も `node --test` 一発で済む。
- seam 分離（純粋関数 ＝ `lib/`、副作用 ＝ CLI 層）で解析性・修正性・再利用性が上がる。
- frontmatter 解析が単一表現化され、lint と gen が別解釈で食い違う事故が消える。

### Negative {#consequences-negative}

- Markdown 文法を拡張するたびに、自前で実装し回帰テストを書く保守コストが要る。
- CommonMark 準拠ではない（意図的な制約付き部分集合）。外部の Markdown ツールとの完全互換は前提にしない。
- 受容した既知の制約が残る：リスト1段ネストのみ／`**外*入れ子*外**`／生 HTML ブロックは空行まで（リッチ図は `=html` フェンスで逃がす）。

### Neutral {#consequences-neutral}

- 出力 HTML は不変（リファクタは挙動保存）。viewer・デザインシステム（`adr/system/`）は変更不要。
- テストの実行形式は `node --test`（引数なし自動探索）。`node --test tools/`（末尾ディレクトリ指定）は Node v25 ではディレクトリを実行モジュール扱いして失敗するため使わない。

## この決定が間違いになるとしたら、何が原因か {#falsification}

> [!WARNING] 前提が崩れたら再検討するトリガー
> 本決定は「利用形態は個人・ローカル」「文法は意図的に小さいまま」「試験性は node:test で足りる」を前提に置く。次が真ならゼロ依存維持は最適でなくなる：

- **制約付き文法を超える要求が複数回出る場合** — 脚注・多段ネスト・定義リスト・表の整列/結合などの要望が **複数回** 挙がるなら、コモディティ部分（Markdown 解析）を成熟ライブラリへ移す（Option B）。*観測指標*: 「この記法が書けない」という要望の再発回数。
- **テスト導入後もパーサのバグ率が高止まりする場合** — 回帰テストを足してもエッジケースの取りこぼしが続くなら、自作解析の限界。*観測指標*: パーサ起因の修正コミット頻度。
- **利用形態がチーム・公開へ変わる場合** — 複数人・OS バラつき・HTML を外部公開へ移ると、**移植性・供給網・ビルド再現性** の重みが上がり、SSG や CI 前提の構成（Option C 方向）が正解に転じうる。*観測指標*: clone して実行する人数、公開要件の発生。
- **`node:test` だけでは試験性が足りなくなる場合** — スナップショット/プロパティテスト等が必要になり標準だけで賄えないなら、テスト用途に限った依存導入を判断する（実行時依存とは分けて検討）。

## Compliance & Monitoring {#compliance}

- **承認前にテストを通す**：ADR ツールへ変更を加えたら `node --test` を緑にしてから承認する。記法を増やすときは **回帰テストを先に書く**（テストファースト）。
- **ゼロ依存の維持**：`tools/` 配下は `node:*` 以外を import しない。実行時依存の追加は打診の上、本 ADR の撤回トリガーに照らして判断する。
- **採番変更時の検証範囲**：ADR 番号を変えたら `grep -rn` の対象に必ず `tools/`（コメント・例示を含む）を入れる。今回の検証漏れ（`adr/` だけ見て `tools/` の旧番号を見落とした）の再発防止。
- **seam の維持**：変換ロジックは `lib/adr-render.mjs` の純粋関数に置き、ファイル IO・引数解析は CLI 層に留める（試験性の継ぎ目を壊さない）。
