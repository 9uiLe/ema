# ADR 開発ワークフロー 検討まとめ & 次やること

## 背景・解決したい課題

- HTML で書かれた ADR（Architecture Decision Records）を、ローカルで開発・修正しながら快適に確認したい。
- ADR を **Web サイトとして公開していない**ため、ローカルでしか確認できず参照しづらい。
- 当初は「ターミナル上で HTML/CSS をレンダリングするツールを自作する」案からスタートしたが、前提を整理した結果、課題の本質が見えてきた。

## なぜ HTML を使っているか（確定した前提）

- **人間**はレビューと意思決定のために ADR を読む。構造化・言語化され、ドメイン/技術知識に紐付けしやすい形を好む。
- **AI** も現状 HTML を読む形で運用している。
- Markdown を試したところ「なんとなくわかった感」が出てしまい、**承認後に「なんか違う」となることが多かった**。これが HTML を選んでいる主因。

## 検討で得られた結論

### 1. 「なんとなくわかった感」の正体
- 処理流暢性（processing fluency）/ 説明深度の錯覚。Markdown の散文はスラスラ読めるため「読みやすさ」を「理解」と取り違え、論理の飛躍や前提の曖昧さを見落としたまま承認してしまう。
- **スラスラ読めること自体がレビューの罠**になっていた。観察は的確。

### 2. HTML が効いていた理由（フォーマットの力ではなく副産物）
- 構造の明示：埋まっていないセクションや薄い論拠が「見えて」しまう。
- 読む際の摩擦：diff をスキャンせず、文書としてちゃんと向き合う。
- 知識への紐付け：リンク・アンカー・表で技術/ドメイン知識に接続できる。

→ 本当に欲しいのは「HTML」ではなく **流し読みを許さず・論理の穴が露出し・根拠が知識に接続される文書**。HTML はそれをたまたま満たしていた。

### 3. レビュー品質（効いている要素）と HTML の付随コストは別物
- **効いている要素**（フォーマット非依存で再現可能）：固定スロットのテンプレート、論拠を強制する欄。
- **付随コスト**：トークン約3倍、diff が汚い、ポータビリティ低、ローカルでしか見られない。

### 4. ローカル制約は表示レイヤーで解ける
- 公開しなくてもリッチに見る手段はある（ターミナルの ANSI レンダリング、ローカルサーバでブラウザ表示、ライブリロード）。
- **表示の問題を理由に保存形式を決めない。**

## 採用する方針

**当面は HTML 正本を維持し、以下2つに取り組む：**

1. ローカル表示ツールを整える（ターミナル＋ブラウザ、編集中ライブリロード付き）。
2. 品質の本体である「流し読みを防ぐテンプレート＋レビュー観点」を明文化・制度化する。

→ 当初やりたかった「ターミナルで HTML を見るツール」は的外れではなく、正しい課題への正しい対処だったと確認できた。

### 将来の検討候補：AsciiDoc
- 技術文書向け。NOTE/WARNING 等の注意ブロック（読み飛ばし防止）、堅牢なクロスリファレンス、include による知識接続を持ちつつ、プレーンテキストで diff がきれい・トークンも軽い。
- 「構造化・言語化・知識紐付け・流し読み防止」の要件に HTML より素直に合致。HTML 変換も標準対応。
- 乗り換えコストはあるため必須ではないが、HTML と Markdown の中間の最有力。

## 次やること

### 着手の分岐（A / B のどちらか、または両方）

**A. ローカル表示ツールを組む**
- 対象：現状の HTML ADR。
- ターミナル表示：`bun ./adr.html` 相当の ANSI レンダリング、または `w3m` / `pandoc + glow`。
- ブラウザ表示：一時ローカルサーバ（`bunx serve` 等）。
- 編集中ライブリロード：`entr` でファイル監視 → 再描画。
  ```bash
  ls path/to/adr/*.html | entr -c sh -c '<レンダリングコマンド> /_'
  ```
- 使い分け：レビュー会＝ブラウザのリッチ表示／開発フロー中の参照＝ターミナル。
- **必要なインプット**：手元の ADR の HTML サンプル1つ（これに合わせて実装する）。

**B. 流し読みを防ぐ ADR テンプレート＋レビュー観点リストを設計**
- テンプレートの固定スロット案：
  - 決定（Decision）
  - 背景（Context）
  - 制約・力学（Constraints / Forces）
  - 検討した選択肢とトレードオフ
  - 却下した案とその理由
  - 結果のプラスとマイナス
  - 前提と、それが崩れる条件
  - **「この決定が間違いになるとしたら何が原因か」欄（論拠を強制）**
- **必要なインプット**：過去に「なんか違う」が起きた具体箇所（前提漏れ／選択肢比較の甘さ／影響範囲の見落とし 等）。これを潰す設計にする。

## オープンな論点
- A と B のどちらから着手するか。
- 将来的に AsciiDoc への移行を本格検討するか。
- AI が ADR を参照する頻度（人間中心か、AI エージェントが繰り返し読むか）— ここの比重で保存形式の最終判断が変わりうる。
